# Decision Brief: 扫描/翻译页总耗时显示 + 设计感 UI

## Recommendation

采用 **布局 A（两行分区）** 的完成摘要卡方案，纯前端实现。卡片在扫描/翻译完成后「变形」替换原有进度区域，展示总耗时与辅助指标，并伴有淡入展开 + 数字递增动画。

## Rationale

- 两个页面复用同一套视觉方案，视觉风格统一
- 纯前端实现（`useRef(performance.now())` 计时），**零后端 Rust 改动**，改动成本最低
- 完成摘要卡比单纯嵌入进度条更有「设计感」—— 进度条消失、摘要淡入，用户明显感知阶段切换
- 可容纳速度、词典/LLM 细分等更多上下文信息，比纯时间更有信息量
- 延续了已有的 `.scan-progress` 卡片样式体系（白色底面、1px 边框、6px 圆角），不突兀

## Alternatives Considered

- **🅱 融入现有 Stats Grid**：只能用于 Dashboard 页（Jobs 页无 stats-grid），且位置容易被忽略，设计空间有限
- **🅲 进度条内嵌计时**：最克制、不改变布局，但「设计感」最弱，不符合用户要求

## Agreed Design Details

| 维度 | 决策 |
|------|------|
| **布局** | 两行分区：右上角 accent 色时间 + 底行 muted 色辅助指标 |
| **翻译细分** | 显示 📖 词典 / 🤖 LLM / ❌ 失败 细分行 |
| **入场动画** | 淡入 + 向下展开（300ms ease-out）+ 数字递增计数（600ms linear） |
| **扫描页指标** | 📦 模组数 + ⚡ 模组/秒 |
| **翻译页指标** | 📝 翻译条数 + ⚡ 条/秒 + 📖 词典 / 🤖 LLM / ❌ 失败 |
| **数据来源** | 纯前端 `performance.now()` + 已有 `scanSummary.mods` / `entryCounts` |

## Risks & Tradeoffs

- 前端计时不包括后端排队/预热时间，严格来说不是「端到端耗时」。如需后者则必须改 Rust 模型
- 翻译页第二行细分（词典/LLM/失败）在扫描页不适用，需要两个页面各自传入不同的辅助指标
- 卡片在取消/失败状态下的表现尚未敲定

## Open Questions

1. 卡片宽度应与进度条一致，还是全宽？
2. 取消/失败状态下该卡片是否仍然出现？应以何种视觉状态（灰色、红色）展示耗时？
3. 翻译页 `entryCounts` 中 `skip` 条目是否应单独展示？
