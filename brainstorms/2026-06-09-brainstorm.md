# Brainstorm Brief: 资源包打包页面重设计

## Current Leaning

用户对当前打包页面的视觉和交互体验有强烈不满（「丑爆了」），通过四轮结构化访谈，已形成一个清晰的重设计方向，但尚未进入技术实现阶段，因此没有最终决策。

## 推荐的设计方案（Strong Preference）

**设计概念：工业流水线式仪表盘**

| 维度 | 结论 |
|------|------|
| **布局** | 仪表盘一览式，自上而下三层：统计+操作 → 装箱动画/模组列表 → 部署按钮 |
| **视觉主题** | 物流/交付（具体为「流水线包装」），装箱动画为核心动效 |
| **按钮策略** | 只有「重新生成」按钮（自动预生成，无独立预览按钮） |
| **中部区域** | 生成时：装箱动画（条目落入箱子，箱子逐格填满）→ 完成后：全部模组列表（状态图标区分），点击展开详情 |
| **冲突检测** | ❌ 完全砍掉 — 用户认为资源包天然可覆盖，无需检测 |
| **部署方式** | 底部三大按钮平铺：保存本地 / 复制到实例 / 打开文件夹 |
| **完成反馈** | 封箱带效果 + 数字飞入 |
| **状态记忆** | 进入页面自动预生成，结果保持到切换任务前 |

## Alternatives Considered & Why Not Preferred

- **分步向导式**: 被否决 — 用户偏好所有信息一目了然的仪表盘
- **终端日志式**: 被否决 — 视觉调性不符合用户的物流主题偏好
- **弹窗聚焦式**: 被否决 — 用户希望打包体验始终可见
- **就地解决冲突 / 跳转编辑**: 被否决 — 用户认为冲突检测本身就是多余的
- **手动部署 / 后台静默部署**: 被否决 — 用户选择三选一主动菜单

## Risks & Open Questions

- **可用性问题**: 只保留「重新生成」一个按钮，新用户可能不知道做了什么（自动预生成不可见）。需要一个清晰的状态指示器
- **动效复杂性**: 装箱动画在 Web 端的高性能实现（CSS/SVG/Canvas 的权衡未讨论）
- **模组列表规模**: 如果扫描 300+ 模组，虚拟滚动的实现和展开详情的性能

## Unresolved Tradeoffs

- **质量报告的存放位置**: 用户要「打包质量报告」又选了「简洁列表+点击展开」，质量报告的具体呈现形态未最终确定（是嵌入列表行？还是抽屉/弹窗？）
- **「全部模组」的边界**: 显示所有模组（包括无语言文件的）用状态图标区分，但无翻译模组占列表空间是否值得
- **多语言目标**: 假设所有用户对「物流/装箱」的视觉隐喻有共鸣

## Transcript Summary

用户指出打包页面「丑爆了」，要求通过多轮访谈探索重设计方向。四轮访谈覆盖了交互节奏、视觉调性、冲突处理、部署仪式感、信息层级、进度反馈、页面状态持久化、按钮设计、模组列表范围和完成反馈共 10 个维度。最终形成了一个以「工业装箱流水线」为主题的仪表盘式设计概念，砍掉了冲突检测和独立的预览按钮，强调自动预生成和装箱动画。用户对每一轮的四选一问题都给出了明确倾向，设计方向高度一致。
